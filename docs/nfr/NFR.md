# Нефункциональные требования

### 1. Производительность (Performance)
#### Задержки (Latency)
* Среднее время оформления заказа < 200мс на p95 при 20RPS
* Среднее время оформления заказа < 500мс на p99 при 20RPS

### 2. Безопасность (Security)
* Мультифакторная аутентификация
* Все соединения только по HTTPS с TLS 1.3

### 3. Масштабируемость (Scalability)
* Архитектура должна поддерживать горизонтальное масштабирование путем добавления новых узлов без даунтайма
* Приложение должно уметь эффективно утилизировать ресурсы многоядерного процессора без изменений кода

### 4. Доступность (Availability)
* Автоматический перезапуск при падении
* Доступность: 99.9%

#### Расчет
* Все время = 365дн. * 24 ч. = 8760 ч.   
* Простой = 8.76 ч.  (считается время, на протяжении которого система выдает ошибку 5хх)  
> Доступность = (Все время - Простой)/Все время = (8760 - 8.76)/8760 = 0.999 * 100% = 99.9%

### 5. Надежность (Reliability) и отказоустойчивость
* Все время = 1 месяц = 1 * 30 * 24 = 720 * 60 = 43200
* Время простоя = 43.2 мин
* Кол-во отказов = 10
> MTBF = (Все время - время простоя)/кол-во отказов = (43200 - 43.2)/10 = 4315.68 мин / 60 = 71.92 ч.

###  6. Сопровождаемость (Maintainability)
* Система должна централизовано собирать логи и метрики (Prometheus, Grafana)
* Легкость обновления и внесения изменений, преимущественно без даунтайма
* Наличие документации для разработчиков
* Проверка покрытия тестами на 80%

> Все время ремонта = 43.2 мин  
> Кол-во ремонтов = 10  
> MTTR = Все время ремонта/кол-во ремонтов = 43.2 / 10 = 4.32 мин.

### 7. Совместимость (Compatibility)
* Приложение должно исправно функционировать в последних версиях браузеров (Chrome, Safari, Yandex, Firefox)

### 8. Удобство использования (Usability)
* Адаптивный дизайн для мобильных устройств
* Пользователь должен находить нужный раздел не более чем за 3 клика


## Расчеты BOTEC (на салфетке)
1. Для действия "Оформить заказ" (Окно пика 80/20, где 80 приходится в период 4.8 часов)
> Всего запросов в сутки = DAU * 0.2 (кол-во запросов на пользователя в день) = 2000000 * 0.2 = 400000  
> RPS_avg = Всего запросов в сутки / 24ч = 400000 / 24 * 3600 = 5  
> RPS_peek = Всего запросов в сутки * 0.8 (кол-во запросов в период пика) / 4.8 (длительность пика) * 3600 = (400000 * 0.8) / (4.8 * 24 * 3600) = 19  

2. Оценка конкурентности
> Concurrency = RPS * AvgLatency = 19 * (100мс * 0.001) = 2

3. Оценка трафика egress/ingress
> Размер запроса/ответа = 2 Кбайт  
> Gbps = Размер запроса * RPS_peek = (2 Кбайт * 64 * 8) * 19 = 311296 бит = 0,000311296
